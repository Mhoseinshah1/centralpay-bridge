# راهنمای پشتیبان‌گیری و بازیابی (فارسی)

<div dir="rtl">

## پشتیبان‌گیری خودکار

- هر شب ساعت **۰۳:۱۵** به وقت سرور (تایمر systemd:
  `centralpay-backup.timer`)
- محل: `/var/backups/centralpay-bridge/` با دسترسی محدود (0600)
- قالب: `pg_dump --format=custom` (فشرده و قابل بازیابی انتخابی)
- هر فایل پس از ساخت با `pg_restore --list` اعتبارسنجی می‌شود و نشان
  `.ok` می‌گیرد
- نگه‌داری: ۱۴ روز (قابل تغییر با `BACKUP_RETENTION_DAYS` در
  `/etc/centralpay-bridge/centralpay.env`)
- **جدیدترین بکاپ معتبر هرگز حذف نمی‌شود**، حتی اگر قدیمی باشد

## دستورها

<div dir="ltr">

```bash
centralpay backup      # بکاپ فوری با اعتبارسنجی
centralpay backups     # فهرست بکاپ‌ها: زمان، حجم، وضعیت اعتبار
```

</div>

## بازیابی

<div dir="ltr">

```bash
centralpay backups
centralpay restore /var/backups/centralpay-bridge/centralpay-YYYYMMDD-HHMMSS.dump
```

</div>

مراحل بازیابی و محافظت‌ها (از ممیزی پشتیبان‌گیری):

1. فایل باید یک فایل معمولی باشد (پیوند نمادین رد می‌شود)، خالی
   نباشد و امضای قالب `PGDMP` را داشته باشد.
2. قفل انحصاری گرفته می‌شود: هم‌زمانی بکاپ و بازیابی ممکن نیست.
3. آرشیو با `pg_restore --list` اعتبارسنجی می‌شود.
4. **checksum از فایل manifest کنار بکاپ (SHA-256) بررسی می‌شود.**
   بکاپ‌های قدیمی بدون manifest فقط با تأیید تایپی `RESTORE-LEGACY`
   پذیرفته می‌شوند و `--yes` این مرحله را دور نمی‌زند.
5. باید عبارت `RESTORE` را تایپ کنید (یا `--yes` بدهید).
6. پیش از هر تغییری، یک **بکاپ پیشابازیابی** ساخته و اعتبارسنجی
   می‌شود.
7. سرویس‌های api و worker و (در صورت فعال‌بودن) ربات مدیر متوقف
   می‌شوند.
8. پایگاه‌داده بازسازی و بکاپ با `--exit-on-error` بازیابی می‌شود.
9. مهاجرت‌ها اجرا و سپس **بررسی یکپارچگی** انجام می‌شود
   (`db-check --repair-sequences`: نسخهٔ مهاجرت، یکتایی شناسه‌ها،
   رویدادهای بی‌صاحب، وضعیت‌های نامعتبر، و موقعیت sequenceها).
10. فقط پس از موفقیت همهٔ مراحل، سرویس‌ها با بررسی سلامت راه‌اندازی
    می‌شوند.
11. فایل بکاپ ورودی حذف نمی‌شود.

اگر بازیابی وسط راه شکست بخورد، **سرویس‌ها عمداً خاموش می‌مانند** —
هرگز روی پایگاه‌دادهٔ نیمه‌بازیابی‌شده سرویس راه‌اندازی نمی‌شود — و
دستورهای دقیق برگشت (بازیابی بکاپ پیشابازیابی) چاپ می‌شود.

## فایل manifest و بکاپ‌های قدیمی

هر بکاپ جدید یک فایل `‎.manifest` کنار خود دارد (بدون هیچ رمز یا
دادهٔ پرداخت): نام فایل، SHA-256، اندازه، زمان ساخت، نسخهٔ برنامه،
نسخهٔ PostgreSQL و نسخهٔ مهاجرت. این فایل به‌صورت اتمیک نوشته می‌شود
و هنگام بازیابی بررسی می‌شود. بکاپ‌های قدیمیِ بدون manifest در خروجی
`centralpay backups` با برچسب `OK-legacy` نشان داده می‌شوند.

## بررسی یکپارچگی دستی

<div dir="ltr">

```bash
centralpay db-check                      # فقط‌خواندنی
centralpay db-check --repair-sequences   # جبران عقب‌ماندگی sequence
```

</div>

## RPO و RTO و بازیابی از فاجعه

- **RPO (حداکثر دادهٔ ازدست‌رفتنی):** با بکاپ روزانهٔ ۰۳:۱۵، تا ۲۴
  ساعت. برای RPO کمتر، `centralpay backup` را زودبه‌زود اجرا کنید.
- **RTO (زمان بازیابی):** در حد چند دقیقه برای بازیابی محلی (اندازهٔ
  پایگاه‌داده + راه‌اندازی سرویس‌ها).
- **هشدار مهم: بکاپ محلی روی همان سرور، بازیابی از فاجعه نیست.**
  اگر خود سرور از بین برود، بکاپ‌های محلی هم از بین می‌روند. انتقال
  دوره‌ای بکاپ‌ها به مقصدی خارج از سرور هنوز خودکار **نیست** و باید
  دستی انجام شود (مثلاً `rsync` به میزبان دیگر). این محدودیت در
  `RELEASE_RISK_REGISTER.md` (مورد ۲۱) ثبت است.

## وضعیت زمان‌بند systemd

<div dir="ltr">

```bash
systemctl status centralpay-backup.timer
systemctl list-timers centralpay-backup.timer
journalctl -u centralpay-backup.service -n 50
```

</div>

زمان اجرا (۰۳:۱۵) به وقت محلی سرور است؛ `Persistent=true` اجرای
عقب‌افتاده پس از خاموشی را جبران می‌کند.

## توصیه‌ها

- بکاپ‌ها را به‌صورت دوره‌ای به یک مقصد خارج از سرور منتقل کنید
  (این کار هنوز خودکار نیست).
- پس از هر بازیابی، `centralpay status`، خروجی `centralpay db-check`
  و چند پرداخت آزمایشی را بررسی کنید.
